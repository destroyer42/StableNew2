PR-CORE-006 - Frame Interpolation and Flicker Reduction

Status: Specification
Priority: MEDIUM
Effort: SMALL
Phase: Post-Unification Core Refinement
Date: 2026-03-29

Context & Motivation

Generated videos may exhibit flickering or choppy motion when frames are rendered only at key points. The current SVD post‑process includes an optional RIFE interpolation stage to insert intermediate frames. However, this functionality is hidden inside the SVD pipeline and not accessible for other workflows (e.g. AnimateDiff output) or configurable by users. Additionally, there is no fallback linear interpolation for environments where the RIFE binary is unavailable. To improve video smoothness across the system, this PR introduces a Frame Interpolation Module that exposes interpolation as a standalone utility and refines flicker reduction.

Goals & Non‑Goals
Goal: Factor out interpolation logic into a reusable module (frame_interpolator.py) that can be called from any video workflow.
Goal: Provide two interpolation modes: a high‑quality mode using RIFE (ncnn/vulkan) when available, and a fallback simple interpolation using image morphing or frame duplication for environments without RIFE.
Goal: Expose interpolation settings to users via the GUI and config (e.g. enable/disable, multiplier).
Goal: Ensure that the interpolation step can be applied to clips generated by other backends (AnimateDiff, Comfy) prior to stitching.
Non‑Goal: Do not reimplement or train interpolation networks; rely on existing RIFE executables. Do not build a fully general video editing suite.
Guardrails
Keep the existing RIFE integration in svd_postprocess.py but refactor it to call the new module; maintain backward compatibility.
The fallback interpolation must be deterministic and not degrade quality when the high‑quality mode is unavailable.
Do not force interpolation; leave it optional in all workflows to avoid unnecessary runtime.
Allowed Files
Files to Create
src/video/frame_interpolator.py – exposes interpolate_frames(frames: list[Image], multiplier: int, method: str) where method can be rife or linear. Includes logic to locate RIFE executable (as in current post‑process) and call it, or perform simple linear interpolation by blending frames.
tests/video/test_frame_interpolator.py – unit tests using small dummy images to verify the number of output frames and method dispatch.
Files to Modify
src/video/svd_postprocess.py – refactor interpolation stage to delegate to frame_interpolator.interpolate_frames instead of running its own RIFE invocation. Preserve existing CLI and config parameters (multiplier, model_dir).
src/video/svd_config.py – ensure interpolation settings (enabled, multiplier, method) are part of the config and passed through to the new module.
src/controller/app_controller.py and GUI – expose interpolation controls (method selection and multiplier) in the UI for SVD and other video tasks.
src/pipeline/pipeline_runner.py – allow interpolation to be applied to other video outputs when configured (e.g. after AnimateDiff generation).
Forbidden Files
Do not modify core diffusion models or secondary motion modules.
Implementation Plan
Implement frame_interpolator.py: Use the logic from svd_postprocess.py to detect the RIFE binary or environment variable. For rife method, create a temporary input/output directory, call RIFE with appropriate arguments, and return the loaded frames. For linear method, perform simple blending: for each pair of frames, create (multiplier-1) intermediate frames by linearly blending pixel values.
Refactor SVD post‑process: Replace the private interpolation routines in svd_postprocess.py (_run_rife_stage and _run_rife_compatibility_stage) with calls to frame_interpolator.interpolate_frames. Preserve existing error handling and configuration options.
Config and UI: Extend SVDConfig.postprocess.interpolation to include a method field with default rife. In the GUI, add a dropdown to choose between rife and linear. Set sensible defaults.
Pipeline integration: Allow interpolation to be invoked after other video generation steps by adding a simple postprocessing hook in pipeline_runner.py. For example, if job_config.postprocess.interpolation.enabled is true, call the frame interpolator on the output frames.
Tests: Write tests for frame_interpolator verifying that the correct number of frames are produced and that fallback method runs when RIFE is unavailable (mocking detection). Write tests for svd_postprocess.py to confirm that interpolation calls the new module.
Manual verification: Run a short video generation with and without interpolation; compare smoothness and flicker. Test both methods and verify that the fallback is acceptable.
Testing Plan
Unit tests for the new interpolator module with small synthetic images. Mock RIFE calls using a dummy executable or monkeypatch.
Integration tests ensure SVD post‑process uses the interpolator and respects the chosen method.
Manual tests generating a clip with interpolation to confirm visual improvement.
Verification Criteria
The frame_interpolator module is reusable and independent of SVD; it can be imported and used in other modules.
SVD post‑process delegates interpolation to the new module without behaviour change for existing users.
Users can choose interpolation method in the UI; when set to linear, the system performs simple blending without requiring external binaries.
All tests pass.
Risk Assessment
Low Risk: The fallback interpolation is simple but acceptable; if RIFE is unavailable the system still produces video.
Medium Risk: Refactoring may introduce regressions in SVD post‑process; mitigated by tests and manual verification.
Tech Debt Analysis

This PR consolidates scattered interpolation logic into a single module, reducing duplication. Deferred work includes experimenting with more advanced interpolation networks or adaptive blending strategies.

Documentation Updates

Update the SVD documentation to describe the new interpolation options and default behaviour. Add a section explaining the difference between RIFE and linear methods and how to install RIFE.

Dependencies

Relies on the existing RIFE binary detection logic and, for fallback, only Python and PIL. No new external dependencies.
