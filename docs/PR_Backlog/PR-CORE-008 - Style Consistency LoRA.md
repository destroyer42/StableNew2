Context & Motivation

StableNew aims to produce a coherent movie from book scenes. While character embeddings (PR‑CORE‑002) ensure facial consistency, there is no mechanism to enforce a uniform visual style (palette, lighting, texture) across scenes. Currently the PromptPack builder merely concatenates user prompts, leaving style entirely up to ad‑hoc prompt wording. The research report identifies a “style consistency LoRA” as missing
newtab
. LoRA models can encode an artistic style (cyberpunk, painterly, watercolor, etc.). Loading such a LoRA in the Diffusion pipeline and appending its token to prompts enforces a consistent look across all generated frames. This PR introduces an optional Style LoRA layer, analogous to character LoRAs but applied globally.

Goals & Non‑Goals
Goal: Provide a way to specify a style LoRA file (e.g. cyberpunk.safetensors) at the project or scene level. When set, the style LoRA’s trigger token is appended to every prompt generated by the PromptPack builder.
Goal: Extend the LoRA manager to load and manage style LoRA weights separately from character LoRAs. Ensure style weights are applied before or alongside character LoRAs during inference.
Goal: Add UI controls (checkbox/drop‑down) for selecting a style LoRA in the GUI’s prompt editor or settings panel.
Goal: Implement validation and fallback: if the selected style LoRA file is missing, gracefully disable it and warn the user.
Non‑Goal: This PR does not implement automatic style learning. Training a style LoRA from scratch is deferred to future research (e.g. PR‑CORE‑020). It merely loads existing LoRA files.
Guardrails
Style LoRA application must not override character LoRAs. The system should compose weights: first apply character LoRAs to preserve identities, then apply the style LoRA with a separate weight multiplier.
Style LoRAs are optional. If no style is selected, prompts remain unchanged and the pipeline falls back to base models.
The style selection UI should clearly indicate that mixing incompatible LoRAs (e.g. a style LoRA trained on a different base model) may yield errors; validation should attempt to detect base model mismatches.
All new features must respect the architecture contract: StableNew owns the intent/config layer【2†L18-L21】, while the backend performs model loading【63†L65-L68】. Loading of style LoRAs happens in the pipeline layer, not in UI code.
Allowed Files
Files to Create
src/training/style_lora_manager.py – defines a StyleLoRAManager class responsible for loading style LoRA weights, validating compatibility, and exposing trigger tokens. This module can reuse logic from character LoRA manager but remains distinct to avoid confusion.
src/config/style_lora_config.py – defines configuration data classes or structures for style LoRA selection (e.g. default style, available styles). May load from a YAML/JSON file listing available style tokens and file paths.
tests/training/test_style_lora_manager.py – unit tests verifying that style LoRAs can be loaded, that trigger tokens are retrieved, and that invalid files raise errors.
Files to Modify
src/services/prompt_pack.py – augment the PromptPack builder to accept an optional style_lora field in the config. When present, append the style token (e.g. <cyberpunkStyle>) to each prompt string, after character tokens and before negative prompts. Also propagate the selected style file into the stage config so the backend can load it.
src/pipeline/job_builder_v2.py – update job builder to include the style LoRA in lora_tags or equivalent list when constructing the NormalizedJobRecord. Ensure style LoRA order does not conflict with character LoRAs.
src/gui/views/prompt_editor_frame.py (or appropriate UI file) – add a field to select a style from the list of available style LoRAs defined in style_lora_config.py. Persist the selection in app settings if desired.
src/pipeline/config_contract_v26.py – extend the schema to allow a style_lora property in the intent layer, referencing a style ID or file name.
Forbidden Files
Do not modify backend model-loading code (e.g. inside src/video/ or Comfy adapters); style LoRA loading must be orchestrated via the pipeline and config layer.
Do not embed style LoRA file content in the repository; only references to file paths and tokens should be stored in config.
Do not hardcode style names in multiple places; maintain a single source of truth in style_lora_config.py.
Implementation Plan
Design style configuration: Create style_lora_config.py to define a StyleLoraDefinition dataclass with fields: name, file_path, trigger_token, weight. Provide a load_style_loras() function that reads from a YAML/JSON file (e.g. data/style_loras.yaml) listing available styles. Expose a get_style(name) helper.
Implement StyleLoRAManager: This class should provide methods to load a style LoRA weight (e.g. using diffusers’ LoRA injection utilities) given a model and to return the associated token. The manager should ensure the base model used for the style LoRA matches the pipeline’s model (to avoid mismatches). If a mismatch is detected, log a warning and skip loading.
Integrate into PromptPack builder: Modify prompt_pack.py so that when building each prompt, it checks for a style_lora entry in the prompt pack metadata. If present, call get_style(name) to retrieve the trigger token and append it to the positive prompt. Also pass style_lora into the stage config so the backend can load the LoRA weight.
Job builder updates: In job_builder_v2.py, add logic to include style LoRA weights and tokens into the lora_tags list used by the pipeline. Ensure the order is [character_loras, style_lora]. Update the config contract to include the style_lora field.
GUI enhancements: Extend the prompt editor panel with a drop‑down list of style names loaded from style_lora_config. When the user selects a style, store this selection in the prompt pack and show the style token in a preview. Provide a default “None” option.
Tests: Implement unit tests for the style manager, verifying that invalid file paths or mismatched base models raise exceptions. Integration tests should confirm that specifying a style in a PromptPack results in the style token being present in the final prompt and that the job builder includes the style LoRA tag in the job record.
Fallback and error handling: If the style file cannot be found at runtime, log a warning and proceed without it. Provide clear error messages in the UI when the user selects an unavailable style.
Testing Plan
Unit Tests: In tests/training/test_style_lora_manager.py, create dummy style definitions and use monkeypatching to simulate loading. Verify that StyleLoRAManager.get_style('cyberpunk') returns the expected token and that missing definitions raise KeyError or custom exceptions.
Integration Tests: Write a test that constructs a PromptPack with style_lora='cyberpunk', runs it through the builder, and checks that the final prompt string contains the style token and that the job record’s lora_tags list includes the style file path. Mock any heavy model loading to avoid GPU requirements.
GUI Tests: If GUI testing infrastructure exists, simulate selecting a style in the prompt editor and verify that the selection persists and affects the underlying PromptPack.
Verification Criteria
A valid data/style_loras.yaml (or similar) is loaded at startup, listing available style LoRAs with names, file paths, and tokens.
Selecting a style in the GUI adds the corresponding token to all prompts and ensures the style file is loaded in the pipeline.
Character LoRA application is unaffected; style LoRAs are applied in addition to character LoRAs. There should be no conflicts or order issues.
All new unit and integration tests pass. Tests for missing files or mismatched base models produce informative errors or warnings.
Config schema updates and default behaviour (no style) remain backwards compatible.
Risk Assessment
Medium risk: Loading additional LoRA weights increases memory use and may slightly slow inference. Mitigate by making style LoRAs optional and documenting GPU requirements.
Compatibility risk: Some style LoRAs are trained on different base models; mismatches will produce artefacts. Provide validation and warnings.
Usability risk: Too many style choices could confuse users. Limit the initial list to a handful of curated styles, and document how to add new ones.
Tech Debt Analysis

This feature introduces a new configuration layer for style LoRAs. To avoid duplication, ensure that style definitions are maintained in a single YAML/JSON file and loaded by both the GUI and backend. Future expansions might include style presets tied to specific scene types (e.g. battle scenes use a “dark gritty” style by default). Deferred work includes adding a tool to train style LoRAs from custom art collections (see PR‑CORE‑020 for research).
